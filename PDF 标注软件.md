# PDF 标注工具需求文档

## 1. 文档目标

设计一款面向科研论文信息抽取的 PDF 标注工具。工具需支持用户在 PDF 页面上框选内容、填写结构化字段、校验数据完整性，并导出为符合目标 JSON Schema 的结果，作为后续知识抽取、数据治理和模型训练的数据来源。

本文档聚焦以下内容：

- 标注任务的操作流程
- 页面与交互需求
- 标注数据模型
- 导出 JSON 的结构与约束
- 非功能性要求

## 2. 使用场景

目标用户：

- 材料领域论文数据标注员
- 课题组研究人员
- 数据治理或知识图谱构建人员

典型任务：

- 打开一篇 PDF 论文
- 在页面上框选标题、正文、图、表、工艺参数、性能数据等证据区域
- 将证据区域映射到目标 Schema 中的具体字段
- 填写字段值、单位、备注、置信度与来源页码
- 导出结构化 JSON

## 3. 产品目标

工具应同时满足两类输出：

1. 页面级标注结果
用于保留原始证据位置，支持追溯和复核。

2. 文档级结构化结果
用于生成符合目标 JSON Schema 的字段化输出。

因此，系统内应区分两层数据：

- `annotation`：页面上的框选证据
- `schema_value`：映射到目标 JSON 字段的结构化值

## 4. 总体流程

推荐的标注流程如下：

1. 用户导入 PDF 文件。
2. 系统渲染 PDF，支持分页浏览和缩放。
3. 用户在当前页框选一个区域。
4. 系统弹出或激活右侧属性面板。
5. 用户选择该区域对应的标签类别。
6. 用户选择该区域映射到的目标 Schema 字段路径。
7. 用户填写字段值、单位、数据类型、备注等信息。
8. 用户保存标注。
9. 系统将该框选区域加入“标注列表”，并同步更新文档级 JSON 草稿。
10. 用户重复操作，直到完成整篇论文标注。
11. 用户执行校验，检查缺失字段、类型错误、枚举值错误和重复冲突。
12. 用户导出 JSON。

## 5. 功能需求

### 5.1 PDF 浏览

系统应支持：

- 本地选择 PDF 文件并加载
- 显示当前页码和总页数
- 上一页、下一页跳转
- 指定页码跳转
- 页面缩放
- 缩放后标注框位置保持正确
- 重新打开同一任务时恢复浏览位置

### 5.2 区域框选

系统应支持：

- 鼠标拖拽框选矩形区域
- 框选后显示高亮边框和半透明填充
- 框选最小尺寸限制，避免误触
- 删除标注
- 编辑已有标注
- 点击列表项后高亮对应页面区域
- 点击页面区域后定位到右侧表单

建议后续预留：

- 多框合并为一个字段证据
- 单个字段关联多个证据框
- 支持非矩形区域或文本行级选择

### 5.3 标签体系

工具需支持两套标签体系同时存在：

1. 页面内容标签
用于描述视觉内容类型，例如：

- 标题
- 正文
- 图
- 图注
- 表
- 表注
- 工艺
- 组织结构
- 性能
- 成分
- 参考文献
- 其他

2. Schema 字段路径
用于描述该标注最终写入 JSON 的位置，例如：

- `metadata.title`
- `metadata.doi`
- `processing.synthesis_methods[].method`
- `properties.mechanical.tensile_properties.yield_strength.value`
- `microstructure.phases_present[].phase_name`

页面内容标签用于辅助人工标注；Schema 字段路径用于生成最终结构化数据。两者不能互相替代。

### 5.4 标注属性面板

每个标注至少包含以下可编辑属性：

- 标注 ID
- 页码
- 区域坐标
- 页面内容一级标签
- 页面内容二级标签
- Schema 字段路径
- 字段值
- 单位
- 原文摘录
- 备注
- 置信度
- 是否已复核

其中：

- “原文摘录”用于保留证据文本，可由用户手动录入
- “字段值”用于写入结构化 JSON
- “单位”应独立保存，避免把数值和单位混为一个字符串

### 5.5 标注列表与任务管理

系统应提供标注列表，至少支持：

- 按页码查看
- 按标签筛选
- 按 Schema 字段筛选
- 搜索关键字
- 删除
- 编辑
- 标记已完成/待复核

系统应提供当前文档任务状态：

- 未开始
- 标注中
- 已完成
- 已复核

### 5.6 Schema 映射与表单生成

系统需要基于目标 JSON Schema 生成可选字段树，并支持：

- 展示层级路径
- 区分对象、数组、标量字段
- 对枚举字段提供下拉选择
- 对布尔字段提供 true/false 选择
- 对数值字段限制输入类型
- 对数组项支持“新增一组”

例如：

- `processing.synthesis_methods[]` 可新增多条工艺记录
- `microstructure.phases_present[]` 可新增多个相
- `characterization_methods[]` 可新增多个表征方法

### 5.7 自动校验

导出前系统应执行至少以下校验：

- 必填字段是否为空
- 数值字段是否为合法数字
- 枚举值是否在允许范围内
- 布尔值是否规范
- 单位是否与字段要求匹配
- 同一字段是否存在冲突值
- 数组对象是否缺少核心子字段

建议设置两级校验结果：

- 错误：阻止导出
- 警告：允许导出，但需提示用户确认

## 6. 数据模型需求

## 6.1 标注对象

每个页面框选区域应至少保存如下信息：

```json
{
  "id": "anno_001",
  "page": 3,
  "bbox": {
    "relX": 0.12,
    "relY": 0.28,
    "relW": 0.34,
    "relH": 0.06
  },
  "content_label": {
    "primary": "性能",
    "secondary": "力学性能"
  },
  "schema_binding": {
    "field_path": "properties.mechanical.tensile_properties.yield_strength.value",
    "array_context": null
  },
  "value": {
    "raw_text": "Yield strength was 512 MPa",
    "normalized": 512,
    "unit": "MPa",
    "value_type": "number"
  },
  "note": "",
  "confidence": 0.95,
  "reviewed": false
}
```

说明：

- `bbox` 使用相对坐标，避免缩放导致位置失真
- `raw_text` 是原始证据文本
- `normalized` 是结构化值
- `field_path` 表示最终写入 Schema 的目标路径

## 6.2 文档级导出对象

导出的 JSON 应至少包含两个顶层部分：

```json
{
  "metadata": {},
  "processing": {},
  "microstructure": {},
  "properties": {},
  "performance": {},
  "characterization_methods": [],
  "computational_details": {},
  "relationships": {},
  "application_context": {},
  "others": {},
  "_annotations": []
}
```

其中：

- 正式业务数据写入目标 Schema 对应字段
- `_annotations` 用于保存所有证据级标注，便于追溯

如果后续要求“严格兼容原 Schema”，则可以提供两种导出模式：

- 标准导出：仅输出 Schema 中定义的正式字段
- 扩展导出：在正式字段外增加 `_annotations`、`_audit` 等扩展字段

## 7. 导出 JSON 需求

### 7.1 导出内容

导出 JSON 应包含：

- 文档级结构化结果
- 每个字段的来源证据
- 页码和坐标信息
- 标注员信息与时间戳
- 校验状态

建议导出结构如下：

```json
{
  "document_id": "paper_001",
  "source_file": "example.pdf",
  "schema_version": "v1",
  "exported_at": "2026-03-10T15:30:00+08:00",
  "annotator": "user_a",
  "review_status": "in_review",
  "data": {
    "metadata": {},
    "processing": {},
    "microstructure": {},
    "properties": {},
    "performance": {},
    "characterization_methods": [],
    "computational_details": {},
    "relationships": {},
    "application_context": {},
    "others": {}
  },
  "annotations": []
}
```

### 7.2 字段写入规则

导出时应遵循以下规则：

- 普通标量字段直接写值
- 数值加单位字段写入 `{ "value": number, "unit": "..." }`
- 枚举字段必须输出为 Schema 允许值
- 数组字段必须输出为数组，即使只有一个元素
- 不存在的数据可留空、置 `null` 或不输出，需在实现阶段统一策略

建议默认策略：

- 已标注字段输出具体值
- 未标注字段不输出
- 导出时可选“补全空字段模板”

### 7.3 证据追溯规则

每个结构化字段建议具备来源追溯能力。实现方式有两种：

1. 在 `annotations` 中维护 `field_path` 关联
2. 在字段旁增加 `_source` 信息

推荐优先方案：

- 正式数据保持干净
- 所有来源追溯统一放在 `annotations`

原因：

- 更容易保持对目标 Schema 的兼容性
- 不会污染主业务字段

### 7.4 文件命名

导出文件命名建议：

- `{pdf文件名}.annotations.json`
- 或 `{document_id}.json`

## 8. 关键交互要求

### 8.1 单次标注操作

单次标注建议交互如下：

1. 用户框选区域。
2. 右侧面板自动显示“新建标注”状态。
3. 用户选择页面内容标签。
4. 用户选择 Schema 字段路径。
5. 用户填写值、单位、原文摘录和备注。
6. 用户点击保存。
7. 系统将标注写入列表，并在画布上保留高亮框。

### 8.2 编辑操作

用户点击已有标注后，系统应：

- 跳转到对应页
- 高亮该框
- 右侧表单回填原值
- 支持修改后保存

### 8.3 导出操作

用户点击“导出 JSON”后，系统应：

1. 执行校验。
2. 展示错误与警告列表。
3. 若存在错误，则阻止导出。
4. 若仅有警告，则允许用户确认继续。
5. 生成 JSON 文件并下载。

## 9. 非功能需求

系统应满足：

- 在本地浏览器环境可运行
- 对常见论文 PDF 页数可流畅翻页
- 缩放与重绘时标注位置不漂移
- 导出结果可被程序稳定解析
- 界面尽量简洁，优先保障标注效率

## 10. 后续版本建议

后续版本可扩展：

- OCR 或 PDF 文本自动提取，减少手录
- 从 Schema 自动生成表单
- 预置材料领域枚举词表
- 批量导入/导出任务
- 多人复核流程
- 标注一致性统计
- 图表专用标注模式
- 自动推荐字段路径

## 11. 本阶段建议的最小可用版本

第一版建议只做最小闭环：

- 导入 PDF
- 分页与缩放
- 矩形框选
- 页面内容标签选择
- Schema 字段路径选择
- 手动填写字段值
- 标注列表管理
- 导出 JSON

第一版暂不强制实现：

- OCR
- 自动抽取
- 多人协作
- 复杂数组编辑器
- 复核工作流

## 12. 结论

该工具本质上不是一个“只做框选”的页面标注器，而是一个“证据标注 + 结构化抽取录入”的混合工具。其核心要求不是仅把页面区域框出来，而是让每个框选结果都能稳定映射到目标 JSON Schema，并在导出时保留证据追溯关系。

后续进入设计或开发阶段时，应优先明确两件事：

- 哪些字段是第一版必须支持的核心字段
- 未标注字段在导出 JSON 中采用何种空值策略
